## Обзор проекта

MyCityQuest — распределённая система, состоящая из трёх независимо развёртываемых слоёв: игрового клиента, административного редактора и серверного API. Слои взаимодействуют исключительно через HTTP API и не имеют прямого доступа к хранилищу данных.

Игровой рантайм и инструменты управления контентом намеренно разделены: изменения в системе создания квестов не затрагивают работающую игру до момента явной публикации. Квесты существуют как версионированные структуры данных, а не как исполняемый код — их интерпретирует движок, встроенный в серверный слой.

Компоненты системы:

- **Game Client** — браузерное приложение на Phaser (HTML5). Читает опубликованные данные квестов через API, отображает игровой интерфейс, записывает прогресс игрока.
- **Admin Editor** — одностраничное приложение на React, доступное только аутентифицированным администраторам. Предоставляет визуальный редактор графа квеста и управление публикациями.
- **Backend API** — Node.js-сервис, единственная точка доступа к базе данных. Выполняет аутентификацию, авторизацию, логику движка квестов и управление версиями.
- **PostgreSQL** — основное хранилище данных: пользователи, квесты, версии квестов, прогресс игроков.
- **Nginx** — обратный прокси, терминирует TLS, маршрутизирует трафик между статическими приложениями и API.

## Архитектурные принципы

### Разделение ответственности

Каждый компонент системы отвечает ровно за одну область:

- **Game Client** отвечает за отображение и пользовательский ввод. Он не содержит логики квестов — он запрашивает текущее состояние у API и отображает результат.
- **Admin Editor** отвечает за создание и редактирование структуры квестов. Он не знает о механике выполнения квестов.
- **Backend API** отвечает за всю бизнес-логику: выполнение шагов квеста, валидацию данных, управление версиями и авторизацию.
- **База данных** отвечает за персистентность и целостность данных.

Это разделение позволяет заменять или модифицировать любой слой без разрушения контракта между остальными, при условии сохранения API-интерфейса.

### Дата-ориентированная архитектура

Квесты хранятся как структурированные данные (JSON-граф), а не как исполняемый код. Поведение квеста определяется декларативно: через типы узлов, условия и ссылки между узлами. Движок квестов является универсальным интерпретатором графа — он не знает о конкретном содержании квеста до момента выполнения.

Это даёт ряд структурных преимуществ:

- Квесты можно создавать, редактировать и валидировать без изменения кода системы.
- Структуру квеста можно проверять статически до публикации.
- Любой квест можно воспроизвести детерминированно, зная начальное состояние контекста.
- Данные квестов переносимы и поддаются версионированию через стандартные механизмы хранения.

### Стратегия версионирования

Каждый квест существует в нескольких версиях. Версия — это неизменяемый снимок графа квеста в момент сохранения. Изменение черновика создаёт новую версию, а не перезаписывает существующую. Опубликованная версия никогда не модифицируется напрямую — любое изменение требует создания нового черновика.

Статусы версии: `draft` → `published` → `archived`. В каждый момент времени не более одной версии квеста может иметь статус `published`. Откат сводится к переводу более ранней версии обратно в статус `published`.

### Границы безопасности

Игровой клиент не имеет и не должен иметь прямого доступа к базе данных. Все операции чтения и записи проходят через API, который применяет авторизацию перед любым взаимодействием с данными. Административный интерфейс физически отделён от игрового: он обслуживается по отдельному пути и защищён отдельной проверкой роли.

Клиент получает только те данные, которые ему необходимы: опубликованный граф квеста и собственный прогресс игрока. Черновики, история версий и данные других пользователей клиенту недоступны.

### Масштабируемость

API-сервис спроектирован как stateless: состояние сессии хранится в JWT, а не на сервере. Это позволяет запускать несколько экземпляров API за балансировщиком нагрузки без разделяемого состояния между экземплярами. База данных остаётся единственным централизованным компонентом и является основным ограничением при горизонтальном масштабировании.

### Изоляция рантайма и системы управления контентом

Редактор квестов и игровой клиент версионируются и развёртываются отдельно. Администратор может работать в редакторе — создавать, редактировать и тестировать черновики — в то время как игроки взаимодействуют с текущей опубликованной версией. Публикация является явной атомарной операцией, а не побочным эффектом сохранения черновика.

## Системная архитектура

### Компоненты и их взаимодействие

```
Браузер игрока                  Браузер администратора
      |                                  |
      | HTTPS                            | HTTPS
      v                                  v
+--------------------------------------------------+
|                    Nginx                         |
|  /         --> /var/www/mycityquest/game          |
|  /admin    --> /var/www/mycityquest/admin         |
|  /api      --> localhost:3000 (Node.js API)       |
+--------------------------------------------------+
                          |
                          | HTTP (internal)
                          v
               +--------------------+
               |   Backend API      |
               |   Node.js/Express  |
               |   port 3000        |
               +--------------------+
                          |
              +-----------+-----------+
              |                       |
              v                       v
     +----------------+      +----------------+
     |   PostgreSQL   |      |   Redis         |
     |   port 5432    |      |   port 6379     |
     |  (основное     |      |  (кэш квестов)  |
     |   хранилище)   |      +----------------+
     +----------------+
```

**Game Client (Phaser)** формируется как статическая сборка (набор HTML/JS/CSS файлов) и раздаётся Nginx напрямую. Во время игровой сессии клиент обращается к API за данными квеста и отправляет обновления прогресса: это единственные сетевые операции. Вся игровая логика, связанная с рантаймом Phaser (анимации, ввод, отрисовка), выполняется локально в браузере.

**Admin Editor (React)** — SPA, также раздаваемое Nginx как статические файлы с пути `/admin`. Редактор реализует графический интерфейс для построения графа квеста и отправляет все мутации состояния напрямую в API. Редактор не хранит постоянного состояния локально — каждое действие синхронизируется с сервером.

**Backend API** — центральный оркестратор. Он получает HTTP-запросы, проверяет JWT, применяет RBAC, исполняет движок квестов и читает/пишет данные в PostgreSQL. API не хранит бизнес-состояние между запросами — всё состояние персистентно в БД.

**PostgreSQL** хранит пользователей, квесты, версии квестов и прогресс игроков. Граф квеста хранится в колонке типа JSONB в таблице `quest_versions`.

**Redis** используется как кэш для опубликованных версий квестов. Кэш инвалидируется при каждой операции публикации.

### Примеры потоков запросов

**Игрок загружает квест:**

```
1. GET /api/quests/:slug/published
   -> Nginx передаёт запрос на localhost:3000
   -> API проверяет JWT из заголовка Authorization
   -> API ищет в Redis кэшированную опубликованную версию
   -> При промахе кэша: запрос к PostgreSQL
      SELECT * FROM quest_versions
      WHERE quest_id = (SELECT id FROM quests WHERE slug = :slug)
        AND status = 'published'
      ORDER BY version_number DESC LIMIT 1
   -> Результат помещается в Redis с TTL
   -> API возвращает граф квеста клиенту

2. GET /api/player/progress/:questId
   -> API проверяет JWT, извлекает user_id
   -> SELECT * FROM player_progress
      WHERE user_id = :userId AND quest_id = :questId
   -> Возвращает current_node_id и context игрока
```

**Администратор публикует квест:**

```
1. POST /api/admin/quests/:id/publish
   -> Nginx передаёт запрос на localhost:3000
   -> API проверяет JWT, проверяет role = 'admin'
   -> Начало транзакции PostgreSQL
      UPDATE quest_versions SET status = 'archived'
        WHERE quest_id = :id AND status = 'published'
      UPDATE quest_versions SET status = 'published', published_at = NOW()
        WHERE id = :versionId
   -> Транзакция фиксируется
   -> Инвалидация кэша Redis для данного квеста
   -> Возвращает обновлённую версию
```

## Движок квестов

### Граф как модель квеста

Квест представлен в виде ориентированного графа, где узлы (nodes) — это дискретные состояния квеста, а рёбра (edges) — переходы между ними. Граф не обязан быть ациклическим: квест может содержать петли (например, повторяющиеся диалоги при неверном выборе). Каждый узел имеет уникальный идентификатор в пределах квеста.

### Типы узлов

| Тип | Назначение |
|-----|-----------|
| `dialogue` | Отображает реплику NPC или описание сцены. Имеет один выходной переход. |
| `choice` | Предлагает игроку набор вариантов. Каждый вариант — отдельный переход к следующему узлу. |
| `condition` | Ветвление без участия игрока. Проверяет флаги или значения в контексте, выбирает один из двух переходов: `true_node` или `false_node`. |
| `flag` | Устанавливает или снимает флаг в контексте игрока. Немедленно переходит к следующему узлу. |
| `item_grant` | Добавляет предмет в инвентарь контекста игрока. Немедленно переходит к следующему узлу. |
| `end` | Терминальный узел. Фиксирует завершение квеста. Исходящих переходов нет. |

### Конечный автомат

Состояние выполнения квеста для конкретного игрока определяется двумя значениями:

- `current_node_id` — идентификатор узла, на котором игрок находится в данный момент.
- `context` — объект произвольной структуры, содержащий флаги, инвентарь и накопленные данные сессии.

Переход состояния:

```
(current_node_id, context) + action → (next_node_id, updated_context)
```

Движок получает текущий узел из графа квеста, обрабатывает его тип, применяет необходимые мутации контекста и возвращает идентификатор следующего узла. Это детерминированная функция: при одинаковых входных данных результат всегда одинаков.

### Асинхронная модель выполнения

Каждый шаг выполнения оформлен как асинхронная операция. Это позволяет узлам инициировать внешние проверки (например, запрос к базе данных для верификации инвентаря) без блокировки. Движок реализован как async/await цепочка: узел разрешается, возвращает переход, следующий узел загружается и разрешается в свою очередь.

Автопереходы (узлы типа `flag`, `item_grant`, `condition`) разрешаются последовательно без ожидания действия игрока, пока не будет достигнут узел, требующий ввода (`dialogue`, `choice`) или `end`.

### Запрет динамического выполнения кода

Движок квестов не поддерживает и не допускает выполнение произвольного кода из данных квеста. Функции `eval`, `new Function()`, `vm.runInContext()` и любые эквиваленты категорически исключены.

Мотивация: квестовые данные создаются в редакторе администратором и сохраняются в базе данных. Если бы данные квеста могли содержать исполняемый код, любая инъекция в хранилище данных привела бы к выполнению произвольного кода на сервере. Все ветвления и условия выражаются через декларативные конструкции (проверка флагов, сравнение значений), интерпретируемые движком через явные ветви кода, а не через eval.

## Структура базы данных

### Таблицы

**`users`** — учётные записи игроков и администраторов.

```sql
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email       VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role        VARCHAR(20) NOT NULL CHECK (role IN ('player', 'admin')),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**`quests`** — метаданные квеста, не зависящие от версии.

```sql
CREATE TABLE quests (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    slug        VARCHAR(100) UNIQUE NOT NULL,
    title       VARCHAR(255) NOT NULL,
    created_by  UUID NOT NULL REFERENCES users(id),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**`quest_versions`** — версионированные снимки графа квеста.

```sql
CREATE TABLE quest_versions (
    id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    quest_id       UUID NOT NULL REFERENCES quests(id) ON DELETE CASCADE,
    version_number INTEGER NOT NULL,
    status         VARCHAR(20) NOT NULL CHECK (status IN ('draft', 'published', 'archived')),
    data           JSONB NOT NULL,
    created_by     UUID NOT NULL REFERENCES users(id),
    published_at   TIMESTAMPTZ,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (quest_id, version_number)
);

CREATE INDEX idx_quest_versions_quest_status ON quest_versions (quest_id, status);
```

**`player_progress`** — текущее состояние прохождения квеста игроком.

```sql
CREATE TABLE player_progress (
    id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    quest_id       UUID NOT NULL REFERENCES quests(id) ON DELETE CASCADE,
    quest_version  INTEGER NOT NULL,
    current_node_id VARCHAR(100) NOT NULL,
    context        JSONB NOT NULL DEFAULT '{}',
    status         VARCHAR(20) NOT NULL CHECK (status IN ('in_progress', 'completed', 'abandoned')),
    started_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (user_id, quest_id)
);

CREATE INDEX idx_player_progress_user ON player_progress (user_id);
CREATE INDEX idx_player_progress_quest ON player_progress (quest_id);
```

### Использование JSONB

Поле `data` в таблице `quest_versions` и поле `context` в `player_progress` имеют тип JSONB по следующим причинам:

- Граф квеста имеет произвольную структуру: количество узлов, их типы и вложенность данных внутри узла неизвестны заранее и различаются от квеста к квесту. Хранение графа в виде реляционных строк потребовало бы сложной нормализации и не дало бы выигрыша в запросах (граф всегда читается целиком).
- JSONB в PostgreSQL поддерживает индексирование через GIN, что позволяет при необходимости делать выборки по содержимому.
- Контекст игрока (`context`) содержит флаги и инвентарь, структура которых определяется конкретным квестом. Фиксированная схема здесь была бы избыточной.
- Оба поля читаются и записываются целиком — нет запросов, обновляющих отдельные атрибуты внутри JSON на уровне SQL.

### Логика версионирования

При каждом сохранении черновика в редакторе создаётся новая строка в `quest_versions` с инкрементированным `version_number` и статусом `draft`. Предыдущий черновик не удаляется.

Публикация выполняется в транзакции:
1. Все строки с `status = 'published'` для данного `quest_id` переводятся в `status = 'archived'`.
2. Целевая строка черновика переводится в `status = 'published'`, устанавливается `published_at`.

Игровой клиент всегда запрашивает версию с `status = 'published'` и максимальным `version_number`. Это гарантирует, что при откате (перевод старой архивной версии обратно в `published`) клиент немедленно начнёт использовать восстановленную версию.

Записанный прогресс игрока сохраняет `quest_version` — номер версии, с которой игрок начал прохождение. Это позволяет завершить прохождение по той же версии графа, даже если квест был опубликован заново с изменениями.

## Процесс публикации

Процесс публикации обеспечивает, что игровая аудитория никогда не видит незавершённый контент и что каждое изменение квеста прослеживаемо.

**1. Создание черновика.** Администратор открывает квест в редакторе. Система автоматически создаёт новую версию со статусом `draft` на основе последней существующей версии. Если квест публикуется впервые — черновик создаётся с `version_number = 1`.

**2. Редактирование и автосохранение.** Все изменения в редакторе сохраняются в поле `data` текущей draft-версии. Сохранение не создаёт новых строк — оно обновляет поле `data` существующего черновика.

**3. Явная фиксация версии.** Администратор может явно зафиксировать версию — это создаёт новую строку `draft` с инкрементированным номером и копией текущего состояния графа. Предыдущий черновик продолжает существовать как архивный снимок.

**4. Публикация.** Администратор инициирует публикацию через кнопку в редакторе. API выполняет атомарную транзакцию перевода статусов (описана в разделе «Структура базы данных»). После публикации кэш Redis для данного квеста инвалидируется.

**5. Доступность в игре.** Следующий запрос игрового клиента на получение данных квеста вернёт новую опубликованную версию. Уже идущие прохождения продолжаются по версии, зафиксированной в `player_progress.quest_version`.

**6. Откат.** При необходимости откатиться к предыдущей версии администратор выбирает архивную версию в истории версий редактора и инициирует её публикацию. Система выполняет ту же транзакцию: текущая опубликованная версия уходит в `archived`, выбранная — переходит в `published`.

## Модель безопасности

### Аутентификация

Система использует JWT (JSON Web Tokens) для аутентификации. При входе пользователь получает два токена:

- **Access token** — короткоживущий (15 минут), передаётся в заголовке `Authorization: Bearer <token>` при каждом запросе к API.
- **Refresh token** — долгоживущий (7 дней), хранится в HTTPOnly cookie, используется для получения нового access token без повторного ввода пароля.

JWT подписывается симметричным ключом (HMAC-SHA256) или асимметричной парой (RS256 — предпочтительно при горизонтальном масштабировании). Payload содержит `user_id`, `role` и стандартные claims (`iat`, `exp`).

### Ролевая авторизация

Роли в системе: `player` и `admin`.

- `player` имеет доступ к: чтению опубликованных квестов, чтению и записи собственного прогресса.
- `admin` имеет доступ к: всему, что доступно `player`, плюс CRUD квестов, управлению версиями, публикации и операциям с пользователями.

Авторизация применяется в middleware API-сервера. Каждый защищённый маршрут декларирует требуемую роль. Middleware извлекает `role` из JWT и отказывает в доступе с кодом `403`, если роль недостаточна.

```
Маршруты /api/admin/* требуют: role = 'admin'
Маршруты /api/player/* требуют: role = 'player' или 'admin'
Маршруты /api/quests/* (GET): доступны аутентифицированным пользователям любой роли
```

### Изоляция административного интерфейса

Admin Editor обслуживается по пути `/admin` и физически представляет собой отдельную сборку. Конфигурация Nginx ограничивает доступ к `/admin` с помощью дополнительного ограничения по IP или базовой аутентификации на уровне прокси (до того как запрос достигнет React-приложения). Двойная защита: Nginx-уровень + JWT-уровень в API.

### Принцип минимальных привилегий клиента

Игровой клиент получает только:
- Данные опубликованной версии квеста (граф без черновиков и архивных версий).
- Собственный прогресс аутентифицированного игрока.

Клиент не может:
- Читать прогресс других игроков.
- Читать черновики или архивные версии квестов.
- Изменять статус версий квестов.
- Обращаться к маршрутам `/api/admin/*`.

Все вышеуказанные ограничения реализованы на уровне API через авторизационные проверки, а не через соглашения на стороне клиента.

### Отсутствие прямого доступа к базе данных

База данных принимает соединения только с `localhost` (или из внутренней Docker-сети). Никакой порт БД не открыт во внешнюю сеть. Единственный компонент, имеющий учётные данные для подключения к PostgreSQL, — это процесс API. Это убирает целый класс угроз: даже при компрометации клиентского кода или токена игрока атакующий не получит прямого доступа к данным.

## Развёртывание

### Топология VPS

Система развёртывается на одном Linux-сервере (Ubuntu 22.04 LTS или аналог). Основные компоненты:

```
/var/www/mycityquest/
├── game/        # Статическая сборка Phaser (index.html + assets)
├── admin/       # Статическая сборка React Admin Editor
└── api/         # Node.js-приложение (исходники или production build)

/etc/nginx/sites-available/mycityquest
/etc/systemd/system/mycityquest-api.service
```

Node.js API-процесс управляется через `systemd`. Сервис настроен на автоматический перезапуск при падении и запускается от непривилегированного пользователя.

### Конфигурация Nginx

Nginx выполняет следующие функции:
- Терминирование TLS (сертификат Let's Encrypt или коммерческий).
- Маршрутизация запросов по пути.
- Отдача статических файлов для `/game` и `/admin` без проксирования.
- Проксирование `/api` на внутренний порт Node.js.
- Принудительный редирект HTTP → HTTPS.
- Заголовки безопасности: `X-Frame-Options`, `X-Content-Type-Options`, `Content-Security-Policy`.

```nginx
server {
    listen 443 ssl;
    server_name mycityquest.example.com;

    location / {
        root /var/www/mycityquest/game;
        try_files $uri $uri/ /index.html;
    }

    location /admin {
        alias /var/www/mycityquest/admin;
        try_files $uri $uri/ /admin/index.html;
    }

    location /api {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Переменные окружения

API-сервис получает конфигурацию исключительно через переменные окружения (не через файлы конфигурации в репозитории):

```
DATABASE_URL      - строка подключения к PostgreSQL
REDIS_URL         - строка подключения к Redis
JWT_SECRET        - ключ подписи JWT (или путь к приватному ключу для RS256)
NODE_ENV          - 'production'
PORT              - порт, на котором слушает API (по умолчанию 3000)
```

### CI/CD

Рекомендуемый процесс деплоя:

1. Сборка фронтенд-артефактов (`game` и `admin`) в CI-среде.
2. Запуск тестов бэкенда.
3. Деплой на VPS: копирование артефактов через `rsync` или `scp`.
4. Выполнение миграций БД: `npm run db:migrate` в директории API.
5. Перезапуск systemd-сервиса: `systemctl reload mycityquest-api`.

Шаги 4 и 5 должны выполняться последовательно: миграции применяются до перезапуска приложения, чтобы новый код работал против уже обновлённой схемы.

## Соображения по масштабируемости

### Горизонтальное масштабирование API

Поскольку API-сервис stateless (состояние сессии в JWT, бизнес-состояние в БД), возможен запуск нескольких экземпляров за балансировщиком нагрузки (например, Nginx upstream с несколькими серверами или облачный load balancer). Единственное требование — все экземпляры должны разделять одинаковый `JWT_SECRET` (или пару ключей при RS256) и иметь доступ к одному и тому же экземпляру PostgreSQL и Redis.

### Стратегия кэширования

Опубликованные данные квестов — тяжёлые для чтения и редко изменяемые объекты — кэшируются в Redis. Ключ кэша: `quest:published:<quest_id>`. Инвалидация происходит при каждой операции публикации или отката.

Прогресс игрока не кэшируется: он обновляется при каждом шаге квеста и должен всегда считываться из БД.

### Индексирование базы данных

Критичные индексы:

```sql
-- Быстрая выборка опубликованной версии квеста
CREATE INDEX idx_quest_versions_quest_status ON quest_versions (quest_id, status);

-- Быстрый поиск квеста по slug
CREATE UNIQUE INDEX idx_quests_slug ON quests (slug);

-- Быстрый доступ к прогрессу игрока
CREATE UNIQUE INDEX idx_player_progress_user_quest ON player_progress (user_id, quest_id);
```

Без этих индексов запросы к `quest_versions` и `player_progress` будут выполнять полное сканирование таблицы при росте объёма данных.

### CDN для игровых ассетов

Статические ассеты игры (изображения сцен, спрайты персонажей, аудиофайлы) составляют основной объём трафика. Их следует выносить на CDN (Cloudflare, Fastly или аналог). Phaser загружает ассеты по URL, сконфигурированным при инициализации загрузчика — достаточно переключить базовый URL ассетов с относительного пути на CDN-домен в конфигурации сборки.

### Пул соединений с базой данных

Node.js API использует пул соединений PostgreSQL (`pg-pool`). Оптимальный размер пула зависит от числа воркеров и характеристик нагрузки; стартовое значение — 10-20 соединений на экземпляр API. При нескольких экземплярах API суммарное число соединений должно оставаться в пределах `max_connections` PostgreSQL.

### Ограничения текущей топологии

Текущая однoсерверная топология создаёт единую точку отказа. При требовании высокой доступности следующим шагом является вынос PostgreSQL на управляемый кластер (например, RDS с read replica) и Redis на управляемый кластер с репликацией.

## Возможные расширения

### Живые обновления квестов

Текущая архитектура требует, чтобы игрок перезагрузил квест для получения обновлённой версии. Альтернатива — WebSocket-канал от API к игровому клиенту. При публикации новой версии квеста сервер рассылает событие `quest:updated` всем активным игрокам, проходящим данный квест. Клиент принимает решение о прерывании или продолжении прохождения.

### Локализация

Текстовые поля в узлах квеста (`text`, `speaker`, `label`) можно расширить до объектов с ключами локалей:

```json
"text": {
  "ru": "Добро пожаловать",
  "en": "Welcome"
}
```

Движок квестов получает `locale` из контекста игрока и при рендеринге выбирает нужное значение. Это не требует хранения отдельных версий квеста для каждого языка.

### Аналитика

Движок квестов может эмитировать события на каждом шаге: `node_visited`, `choice_made`, `quest_completed`, `quest_abandoned`. События передаются в аналитический сервис (через очередь сообщений или прямой HTTP-вызов) за пределами основного потока выполнения запроса. Это даёт данные для построения воронок прохождения и выявления узких мест в дизайне квестов.

### Событийно-управляемые триггеры

Текущая модель предполагает, что квест начинается по явному действию игрока. Расширение: внешние события (геолокационный триггер, временное условие, завершение другого квеста) могут инициировать переход состояния квеста. Это требует отдельного сервиса-диспетчера событий, способного записывать обновления прогресса асинхронно.

### Совместимость с многопользовательским режимом

В текущей архитектуре контекст квеста сугубо индивидуален. Многопользовательские квесты потребуют разделяемого контекста: несколько игроков влияют на состояние одного квеста одновременно. Это вводит конкурентный доступ к записи. Возможные подходы: оптимистичная блокировка с версионированием контекста, CRDT-структуры для конфликтно-устойчивого состояния или событийно-ориентированная модель с одним авторитетным сервером состояния.

### Редактор с историей изменений

Текущий редактор сохраняет черновик как единый снимок. Расширение: операционное преобразование (OT) или CRDT-основанный collaborative editing, позволяющий нескольким администраторам работать с одним квестом одновременно, с сохранением истории каждой правки.
